# クイズ得点表アプリ — デザインリニューアル用 仕様書（バイブコーディング向け）

## 0. この仕様書の使い方

既存アプリ「quiz_point」の**デザインを一新する**ための仕様書。Gemini / Canvas 等の
生成AIに渡して、UI/UX を作り直すことを想定している。

### ⚠️ 最重要の前提：リアルタイム同期にはサーバが要る
本アプリの肝は「ホストが操作すると、全参加者のスマホに即時反映される」こと。これには
バックエンド（WebSocket サーバ）が必須。生成AIが作るのは通常フロントエンドのみなので：

- **方式A（推奨）**：既存の Node + Socket.IO サーバをそのまま使い、**フロントだけ作り直す**。
  → 後述「6. 通信プロトコル（Socket.IO）」に合わせて実装すれば、既存サーバで即動く。
- **方式B**：バックエンドごと作り直す。Firebase Realtime Database / Firestore など
  リアルタイムBaaSを使い、後述「5. データモデル」「7. 得点計算ルール」を再現する。

> まず方式Aで「見た目だけ刷新」するのが圧倒的に早い。生成AIには
> 「以下の Socket.IO イベント仕様に合わせてフロントエンドを作って」と伝える。

## 1. プロダクト概要

クイズ大会用の**リアルタイム得点表**。司会者（ホスト）が得点を操作し、参加者は自分の
スマホで全員の順位・点数をリアルタイムに眺める。

- **ホスト**：1名。部屋を作り、共有リンクを配り、全員の得点を操作する。
- **参加者**：複数名。リンクから名前を入れて入室し、**閲覧のみ**（操作不可）。

特徴：単純な加減算だけでなく **かけ算・わり算（小数・分数）** と
**チーム得点（メンバーの積）** に対応。早押し向けの **○×ワンタップ採点モード** あり。

---

## 2. ユーザーと権限

| 役割 | できること |
|------|-----------|
| ホスト | 部屋作成、得点の加減乗除、一括操作、Undo、チーム編成、参加者の改名/削除 |
| 参加者 | 名前を付けて入室、得点表の閲覧のみ |

- ホスト権限は「部屋作成時に発行されるトークン」をブラウザに保存して識別する
  （＝部屋を作った端末だけがホスト操作できる）。

## 3. 画面一覧

### (A) トップ / ホーム
- タイトル＋説明
- 「部屋を作る（ホスト）」ボタン → 部屋を作成しホスト画面へ
- 「部屋コードで入る」入力欄＋ボタン（6文字コード）

### (B) 参加者：入室フォーム
- 部屋コード表示
- 名前入力欄＋「入室」ボタン
- エラー表示（存在しない部屋／名前未入力）
- 一度入室した端末は次回以降は名前を自動再利用して即入室

- チーム得点ランキング（チームがある時のみ表示）

## 4. 機能詳細

### 4.1 部屋
- ホストが部屋を作成すると **6文字の部屋コード**（紛らわしい文字を除いた英数字）と
  **共有URL**（`/?room=コード`）が発行される。
- 参加者はURLか部屋コードで入室。状態はサーバのメモリに保持（再起動で消える）。

### 4.2 得点操作（通常モード）
各プレイヤー行で：
1. 演算子 **＋ − × ÷** から1つ選ぶ（既定は ×）
2. 値を入力（`2` / `0.5` / `1/2` など、**小数・分数OK**）
3. 「適用」 → その人の得点に演算が適用される

### 4.3 ○×モード（早押し採点）
- トグルでON。「1問の得点」を事前に設定（既定 10、小数・分数可）。
- ONにすると各プレイヤー行が **○（＋規定点）** / **×（−規定点）** の2ボタンに変わる。
- ワンタップで加点・減点。OFFに戻すと通常の ＋−×÷ 操作に戻る。
- 設定（ON/OFF・得点）はホスト端末に保存され次回も維持。

### 4.4 一括操作
- **全員を1に**：全プレイヤーの得点を 1 にする（チーム＝積の起点に便利）。
- **全員0リセット**：全員 0 に（確認ダイアログあり）。
- **Undo（一つ戻す）**：直前の操作を取り消す（履歴は最大50件）。

### 4.5 チーム
- チームの作成・改名・削除。プレイヤーをプルダウンでチームに割当（未所属も可）。
- **チーム得点 = メンバー各得点の「積（かけ算）」**。
  - 例：メンバーが 2, 1, 2, 1, 2 点 → チーム得点 = 2×1×2×1×2 = **8**
- チーム削除時、所属メンバーは「未所属」に戻る。

### 4.6 プレイヤー管理
- 改名、削除（確認あり）。
- 参加者は入室時に作成され、降順でランキング表示。

### 4.7 並び順・強調
- 個人・チームとも **得点の降順**。同点は名前順。
- 1位をハイライト。参加者画面では**自分の行**を別色で強調。

## 5. データモデル

```
Room {
  id: string            // 6文字の部屋コード
  hostToken: string     // ホスト認証用（参加者には渡さない）
  players: Player[]
  teams: Team[]
  history: Snapshot[]   // Undo 用（最大50）
}

Player {
  id: string
  name: string          // 最大30文字
  score: string         // 有理数 "n/d"（後述）
  teamId: string | null
}

Team {
  id: string
  name: string          // 最大30文字
}
```

参加者に配信する公開状態には `hostToken` と `history` は**含めない**。

## 6. 通信プロトコル（Socket.IO）※方式Aで必須

クライアント→サーバはすべて **ack コールバック**で結果を受け取る。
サーバ→クライアントは状態が変わるたびに `state` イベントをブロードキャスト。

### クライアント → サーバ
| イベント | 送信データ | 応答(ack) |
|---------|-----------|-----------|
| `createRoom` | なし | `{ ok, roomId, hostToken }` |
| `hostJoin` | `{ roomId, hostToken }` | `{ ok, state }` / `{ ok:false, error:'unauthorized' }` |
| `joinRoom` | `{ roomId, name, playerId? }` | `{ ok, playerId, state }` / `{ ok:false, error:'room_not_found'\|'invalid_name' }` |
| `checkRoom` | `{ roomId }` | `{ ok: boolean }` |
| `op` | `{ roomId, hostToken, type, payload }` | `{ ok, canUndo }` / `{ ok:false, error }` |

### `op` の type と payload
| type | payload | 意味 |
|------|---------|------|
| `score` | `{ playerId, operator, valueStr }` | 得点に演算適用（operator: `+` `-` `*` `/`） |
| `setScore` | `{ playerId, valueStr }` | 得点を指定値に設定 |
| `setAll` | `{ valueStr }` | 全員を指定値に |
| `resetAll` | `{}` | 全員0に |
| `removePlayer` | `{ playerId }` | 削除 |
| `renamePlayer` | `{ playerId, name }` | 改名 |
| `createTeam` | `{ name }` | チーム作成 |
| `renameTeam` | `{ teamId, name }` | チーム改名 |
| `removeTeam` | `{ teamId }` | チーム削除 |
| `assignTeam` | `{ playerId, teamId\|null }` | チーム割当 |
| `undo` | `{}` | 直前操作を取消 |

`valueStr` は文字列で送る（例 `"2"`, `"0.5"`, `"1/2"`）。サーバ側でパース・検証する。
不正値なら `{ ok:false, error:'invalid_value' }`。

### サーバ → クライアント
| イベント | データ |
|---------|--------|
| `state` | `{ roomId, players:[{id,name,score,teamId}], teams:[{id,name}] }` |

- `score` は **有理数文字列 "n/d"**（例 `"8/1"`, `"1/2"`）。表示・並べ替えはクライアント側で行う（次節）。

## 7. 得点計算ルール（重要・再現必須）

得点は **内部で有理数（分数）として厳密に保持**する。これにより
「÷3 してから ×3」しても誤差なく元に戻る。表現は文字列 `"n/d"`（n:符号付き整数, d:正の整数, 既約）。

- **入力のパース**：`"2"→2/1`、`"0.5"→1/2`、`"-1.5"→-3/2`、`"1/2"→1/2`。
  空文字・記号・0除算は不正として弾く。
- **四則演算**：分数同士で計算し既約化。`/` で分母0は不可。
- **チーム得点**：メンバーの `score` をすべて掛け合わせる（積）。メンバー0人は 0。
- **表示用フォーマット**：整数ならそのまま（`8/1`→`8`）、そうでなければ**小数第2位まで**
  四捨五入し末尾0を除去（`1/2`→`0.5`、`1/3`→`0.33`）。
- **並べ替え**：`n/d` を数値化（n÷d）して降順。同点は名前順。

> 方式B（バックエンド作り直し）の場合、小数の浮動小数点誤差を避けるため
> この有理数方式を踏襲することを強く推奨。少なくとも「表示は小数第2位丸め」を守る。

## 8. UI/UX デザイン方針（ここは自由に刷新してOK）

**作り直してほしいのは主にここ。** 機能は上記を満たしつつ、見た目は刷新する。

必須要件：
- **モバイルファースト**（参加者はスマホ閲覧、ホストもスマホ操作が多い）。
- **日本語UI**。
- **得点が主役**：数字が大きく見やすい。ランキングの順位変動が直感的に分かる。
- **ホストの操作ボタンは押しやすく**（特に ○×モードの ○/× は大きく、色で区別）。
- 1位・自分の行が一目で分かるハイライト。
- リアルタイム更新時に違和感のないスムーズな反映（軽いアニメーションは歓迎）。

自由に決めてよい点：
- 配色・フォント・余白・カード/リストのデザイン・ダーク/ライト。
- 順位変動アニメーション、紙吹雪などの演出。
- アイコン、ロゴ、トーン＆マナー。

現状の不満点（参考）：デザインが地味。→ もっと「クイズ番組」的な高揚感・視認性が欲しい。

## 11. 得点履歴グラフ

得点の推移を**折れ線グラフ**で見せる。1人＝1本の線、横軸は「問題数（ラウンド）」または
「時刻」、縦軸は得点。順位の逆転がひと目で分かるのが狙い。

### 表示
- 参加者・ホスト両方から見られる（タブ or 開閉パネル）。
- 各線に名前ラベル（凡例）。色はプレイヤーごとに固定。
- スマホでも見やすいよう、線の本数が多い時は上位N人＋自分だけ強調する等の配慮があると良い。
- ライブラリは Chart.js / Recharts など任意。軽量なら自前Canvasでも可。

### データの取り方（実装方式の選択）
グラフには「時系列の得点スナップショット」が必要。下記から選ぶ：

- **方式①（バックエンド改修・推奨）**：サーバが部屋ごとに `timeline` を保持。
  得点が変わる操作のたびに「全員の得点スナップショット」を1点追加し、`state` に含めて配信。
  → リロードしても残る／全端末で一致する。下記データ形を追加：
  ```
  Room.timeline: Array<{
    t: number,                     // タイムスタンプ
    label?: string,               // 任意（"第3問" など）
    scores: { [playerId]: "n/d" } // その時点の全員の得点（有理数文字列）
  }>
  ```
  - 「次の問題へ」ボタン（任意）でラウンド区切りラベルを打てると見やすい。
  - 配信時はグラフ用に数値化（n÷d）した値を使ってよい。
- **方式②（バックエンド改修なし）**：新フロントが、受信した `state` の変化を
  **自分のメモリに記録**してグラフ化。サーバ変更ゼロで動くが、その端末が接続して以降の
  データのみ＆リロードで消える。まず手軽に出すならこれ。

> どちらでも見た目は同じにできる。確実さ重視なら①、手軽さ重視なら②。

## 12. 演出・アニメーション

「クイズ番組」的な高揚感を出す。やり過ぎず、得点操作の手応えが上がる程度に。

### 得点・順位の動き
- **得点変動**：変わった行をフラッシュ／ポップ。数字は**カウントアップ**で増減。
  加点は緑系、減点は赤系の一瞬の色づけ。
- **順位変動**：行の並び替えを**スムーズに移動**（FLIPアニメ）。瞬間入れ替えにしない。
- **1位交代**：トップが入れ替わった瞬間に強調演出（王冠🏆／グロー／軽い紙吹雪）。

### ○×採点の手応え（○×モード）
- ○（正解）：緑のフラッシュ＋「ピンポン♪」効果音。
- ×（不正解）：赤の振動（shake）＋「ブッブー」効果音。

### 効果音
- 効果音は任意。必ず**ミュート切替**を用意（既定OFFでもよい）。
- 端末の自動再生制限に配慮（最初のユーザー操作後に有効化）。

### フィナーレ（任意）
- 「結果発表」ボタンや終了時に、1位を大きく表示＋紙吹雪などのお祝い演出。

### 配慮
- アニメは控えめ・短め（数百ms）。過剰な点滅は避ける。
- パフォーマンス：多人数でもカクつかないこと（CSS transform 中心、再描画を最小化）。
